# 2. Services manifest authored by core-platform-dashboard

Date: 2026-04-21

## Status

Proposed

## Context

Upptime's native configuration format (`.upptimerc.yml`) mixes two concerns in a single file: the list of services to probe, and operational settings for the status page itself (branding, theme, notifications, custom domain, navbar). The first is driven by which core-platform applications have opted in; the second is a local concern of this repo.

When PT-397 first shipped, both concerns were hand-edited. The core-platform-dashboard team now maintains the authoritative catalogue of applications (see `core-platform-dashboard` ADR 0008), so adding an application should flow from that catalogue rather than a manual PR against this repo.

Two options were considered for where that boundary sits:

- **Dashboard writes `.upptimerc.yml` directly.** One hop, but the dashboard would need to know about notification schema, branding, navbar, and theme, either preserving them through merges or taking them over. Preserve-and-merge on a hand-edited file is fragile; taking over would move operational concerns into a repo that should not carry them.
- **Dashboard writes a generic services manifest; this repo compiles the Upptime config.** Clean boundary: dashboard says "monitor these services"; this repo decides how.

The second option isolates every Upptime-specific concern to this repo and keeps hand-maintained configuration safe from being overwritten.

## Decision

We will adopt a two-file separation of concerns in this repo:

- `services.generated.yaml` — owned by core-platform-dashboard. Contains the services to monitor. Modified only by PRs opened by the dashboard. Never hand-edited.
- `.upptimerc.yml` — owned by this repo. Contains branding, theme, `status-website` block, notifications, custom domain, and any non-DU services that need manual inclusion. The `sites:` key in this file is regenerated by the compile workflow below.

A GitHub Actions workflow in this repo (`compile-config.yml`) triggers on any push that touches either input. It reads `services.generated.yaml` plus the hand-maintained portion of `.upptimerc.yml`, regenerates the `sites:` key, and commits the result. Upstream Upptime workflows (`uptime.yml`, `static-site.yml`, etc.) continue to consume `.upptimerc.yml` as they always have.

### Contract for `services.generated.yaml`

```yaml
version: 1
services:
  - name: <display-name>
    url: <https url, typically /livez>
    owner: <du owner string>
```

`version` is `1` at this revision. The file is written in its entirety on each dashboard update; this repo treats it as read-only.

### How a new application joins the status page

1. The application owner adds `statusPage: { url: "https://..." }` to the DU descriptor YAML in their own repo (see `core-platform-dashboard` ADR 0008).
2. The next dashboard catalog refresh picks it up.
3. The dashboard opens a PR against this repo updating `services.generated.yaml`.
4. A reviewer merges the PR.
5. `compile-config.yml` regenerates `.upptimerc.yml` on merge.
6. Upptime's scheduled workflows pick up the new site on their next five-minute cycle.

### Manual-maintenance escape hatch

A non-DU service — an external dependency, a partner system, an endpoint that is not represented as a DU — may be added directly to `.upptimerc.yml` in a clearly-marked `# manual-sites` section. The compile workflow must preserve that section when regenerating. This path is an exception and should not be used for anything that can be modelled as a DU.

## Consequences

- Good: application owners opt into the status page by editing their own repo. No PR against this repo is needed for the common case.
- Good: branding, notifications, theme, and tool choice remain local to this repo and are not at risk of being clobbered by dashboard-side automation.
- Good: the PR review surface for a new service is minimal — a single entry under `services:` — so reviewers can focus on "is this service supposed to be public?" rather than on interleaved tool-configuration noise.
- Good: swapping Upptime for another monitoring tool is a change that is local to this repo. The dashboard's contract (`services.generated.yaml`) is unchanged.
- Bad: a new compile step must be maintained in this repo. The step is small (~30 lines of workflow YAML plus a short rendering script) but is a new moving part.
- Bad: a push that edits both `services.generated.yaml` and the hand-maintained portion of `.upptimerc.yml` in one commit requires the compile workflow to merge correctly. The workflow must treat `services.generated.yaml` as authoritative for the `sites:` key and leave everything else alone.
- Bad: if the dashboard is unavailable for an extended period, new applications cannot be added via the standard path. The escape hatch (direct edit of `.upptimerc.yml`) remains available but should not become the default.
